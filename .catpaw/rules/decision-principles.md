# 决策原则

本规则定义决策逻辑修改时必须遵守的不变量和设计约束。

## 1. 优先级铁律

```
偏好零违规 > 避免亏损接单 > 最大化单笔利润 > 最大化步数效率
```

违反顺序的任何修改必须附带明确说明和量化分析。

## 2. 架构不变量

- **SchedulePlanner 最高优先**：如果它返回了强制动作，后续层不得覆盖。
- **RuleEngine 纯确定性**：过滤逻辑不得引入随机性或 LLM 调用。
- **LLM 可选性**：去掉 LLM 后系统仍能正常运行（纯规则模式）。
- **StateTracker 自包含**：状态维护不依赖外部存储或 LLM 记忆。

## 3. 修改检查清单

修改决策逻辑前，确认以下问题：

- [ ] 修改会破坏 SchedulePlanner 的强制动作优先级吗？
- [ ] 修改引入了对特定 driver_id 的硬编码吗？
- [ ] 修改后 LLM 完全不可用时系统还能正常运行吗？
- [ ] 修改影响了 simkit/server 目录下的文件吗？
- [ ] 修改后所有现有测试仍能通过吗？

## 4. 评分修改规范

CargoScorer 的打分公式修改时：
- 必须在 PR/commit message 中说明修改动机
- 必须附带至少一个具体数值推导示例
- 权重参数不得硬编码为 magic number，必须定义为命名常量

## 5. 偏好规则扩展

新增偏好规则处理时：
- 在 `docs/agents/03-preference-rules.md` 中登记规则说明
- 在 SchedulePlanner 中实现强制动作（如果是硬约束）
- 在 RuleEngine 中实现过滤（如果是货源过滤类）
- 添加对应单测覆盖正常和边界场景

## 6. LLM Prompt 修改

修改 StrategyAdvisor 或 PreferenceParser 的 prompt 时：
- 不得在 prompt 中包含裸花括号 `{}`（会与 str.format 冲突）
- JSON 示例中的花括号必须转义为 `{{` 和 `}}`
- 修改后跑一次完整 10 步仿真确认无崩溃
