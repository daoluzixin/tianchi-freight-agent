## 瓶颈一：D004 首单跨天循环（4,000 元）

### 现象

D004 偏好："只要这天接了单，首单开工不得晚于当天中午 12 点"。31 天中 23 天违规，每天 200 元（封顶后 4,000）。

### 根因

Agent 陷入稳定的恶性循环：

```
下午 15:00-17:00 接 300+km 长途单（执行 15-20h）
  → 次日凌晨 02:00-08:00 在偏远地点卸货
  → 附近无货源，反复 query_cargo → wait → query_cargo
  → 12:00 进入午休安静窗口，被强制等到 13:00
  → 13:00-16:00 才找到货并接单 → 违规
  → 这个新单又是长途跨夜单 → 循环重复
```

评测系统判定"首单开工时间"看的是当天**新接**的第一个 `take_order` 的 `action_start`，跨夜订单的完成不算当天首单。

### 代码层面

`cargo_scorer.py` 第 280-298 行有 `first_order_cross_day_penalty`，但惩罚上限 = `200 × min(2.0, overshoot_hours)` = 最多 400 分。长途单毛利 1000+ 元，净评分仍为正，照接不误。

缺失的能力：

1. 无"跨日影响前瞻"——不评估"接这单后明天能否在 12 点前接到新单"
2. 无"宁愿不接单"逃逸——不接单则不违规（偏好是"只要接了单"才检查），但 Agent 没有这个策略
3. `schedule_planner._handle_first_order_deadline` 只能催促 WORK 模式，附近没货时催促无效

### 方案

**方案 A（rule_engine 硬过滤）**：当 `first_order_deadline_hour` 存在时，如果货源的 `finish_estimate` 落在次日 deadline 之后（`finish_day > current_day` 且 `finish_hour_in_day > deadline_hour`），直接过滤。迫使 D004 接短途单。

**方案 B（大幅提升评分惩罚）**：将 `first_order_cross_day_penalty` 从 200 提升到 800-1000，使跨天长途单的净评分为负。比方案 A 更柔性，允许极高利润的长途单通过。

**方案 C（午后不接单逃逸）**：当天已过 deadline 且尚未接单时，如果剩余时间不足以完成一单并在次日 deadline 前再接一单，则当天放弃接单（wait 到次日）。不接单 = 不违规。

预期收益：23 天违规降到 5-8 天，减少 **2,000-3,000 元**。

风险：过滤长途单会降低毛收入，需要观察净效果。

## 瓶颈二：休息跨日切割（D001+D006+D008+D002 共 3,300 元）

### 现象

评测系统按自然日 `[00:00, 24:00)` 裁剪 wait interval，取当天最长连续 wait 段判定是否达标。当 rest 跨越午夜时，两天各分到一半，可能都不达标。

典型案例：

- D001 Day 26 (3/27)：rest 从 16:01 开始，480min 到次日 00:01。当天 16:01→24:00 = **479min**，差 1 分钟
- D001 Day 23 (3/24)：rest 从 16:34 开始，当天只有 **446min**
- D001 Day 0 (3/1)：首日完全没安排 rest

### 根因

`schedule_planner._handle_daily_rest` 输出的 rest 时长本身是正确的（480/300/240min），但**时机选择**有问题——当长途单在 16:00 后完成时，rest 起始点太晚，必然跨越午夜。

评测的裁剪逻辑（`_wait_intervals_for_day`）：

```python
s = max(c["step_start"], day * 1440)
e = min(c["step_end"], (day + 1) * 1440)
```

### 方案

在 `_handle_daily_rest` 中增加"午夜对齐"逻辑：

```python
remaining_today = 1440 - (sim_minutes % 1440)
if remaining_today < min_continuous_rest_minutes:
    # 当天放不下完整 rest，加长 rest 确保裁剪后两天都达标
    rest_duration = min_continuous_rest_minutes + (min_continuous_rest_minutes - remaining_today)
else:
    rest_duration = min_continuous_rest_minutes
```

或者更简单：如果 `remaining_today < rest_needed`，先发一个短 wait 到午夜，再发完整 rest。

首日特殊处理：确保 Day 0 在首步决策时优先安排 rest。

预期收益：减少 **1,500-2,500 元**，覆盖 4 个司机。

风险：加长 rest 减少工作时间，但每次违规 200-400 元，多休息 30-60 分钟的机会成本远低于罚金。

## 瓶颈三：D009 go_home 逻辑缺陷（3,600 元）

### 现象

D009 偏好："每天 23 点前车辆须在自家位置（23.12, 113.28）一公里内"。4 天违规，每次 900 元。

4 次违规明细：

| 日期 | 迟到 | 根因 |
|------|------|------|
| 3/3 | 全夜未归 | 接了跨夜熟货 240646，交付于次日 03:11，当天在韶关 195km 外 |
| 3/11 | 10 min | 距家 53km，出发偏晚 |
| 3/21 | 31 min | _handle_daily_rest 发出 172min 大 wait，跳过 go_home 触发窗口 |
| 3/27 | 1 min | 距家 106km，buffer 不足 |

### 根因

三个子问题叠加：

**问题 1：rest 与 go_home 竞争**。`_handle_go_home` 优先级高于 `_handle_daily_rest`（代码第 87 行 vs 第 138 行），但当 go_home 条件不满足时，daily_rest 可以发出长达 480min 的 wait。这个 wait 执行期间不再检查 go_home，可能直接跳过临界出发时间。

**问题 2：固定 60min buffer 不适配长距离**。`travel_min + 60 >= time_to_deadline` 作为触发条件，对 150km 路程（约 150min）意味着要在 deadline 前 210min 触发。但如果在 210min 之前有一个大 rest，就会跳过窗口。

**问题 3：接单时只检查"卸货当天"的 deadline**。3/3 接熟货后执行 738min 跨天，rule_engine 检查的是 3/4 的 deadline，而 3/3 的回家被忽略了。

### 方案

1. **rest 前加 go_home 前瞻**：在 `_handle_daily_rest` 发出长 wait 前，检查 `wait_end + travel_to_home > deadline`，如果是则先触发 go_home
2. **动态 buffer**：`buffer = max(60, travel_min * 0.3 + 30)`，长距离给更多余量
3. **接单 guard 加交付时间预估**：如果订单执行时间跨越当天的 go_home deadline，且当前距家 > 50km，拒绝该单

预期收益：4 次降到 1-2 次，减少 **1,000-2,000 元**。

风险：提前回家损失接单机会，但每次违规 900 元，值得。

## 瓶颈四：D003 空驶超额（2,000 元）

### 现象

D003 月度空驶限额 100km，实际 302km，超额 202km，罚 2,000 元。

### 根因

大量空驶来自"接单失败"——Agent 空驶去装货点但货已过期（`load_time_window_expired`），空驶白跑但仍计入配额。从 action log 看到 step 36/41/43/45/57/59/73 等多次失败，每次 8-30km 无效空驶。

此外，rule_engine 的空驶配额动态收紧从 50% 用量才开始（50%→25km, 75%→12km, 100%→5km），对 100km 这种极低限额来说收紧太晚。

### 方案

1. **接单前检查 load_time_window**：如果货源的装货窗口即将过期（剩余时间 < 空驶时间 × 1.2），直接跳过
2. **提前收紧空驶配额**：对 100km 以下限额的司机，从 30% 用量就开始收紧
3. **加大评分惩罚**：`deadhead_budget_penalty_mult` 从 2.0 提升到 4.0

预期收益：超额从 202km 降到 50-80km，减少 **500-1,000 元**。

## 瓶颈五：D003 安静窗口跨越（1,400 元）

### 现象

D003 安静窗口 02:00-05:00，正常模式下仍有 7 次违规。

### 根因

长途单的实际执行时间超过了 rule_engine 的估算。当前 safety_margin = `max(30, 10% × 运输时间)`，对 10 小时的长途单只加 60min 余量，但实际偏差可能更大。

### 方案

将 safety_margin 提升到 `max(60, 15% × 运输时间)`，或者对有安静窗口的司机，在窗口前 3 小时停止接预估完成时间跨越窗口的长途单。

预期收益：7 次降到 2-3 次，减少 **500-800 元**。
